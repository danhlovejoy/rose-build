Module 4: Visual Similarity

Readings & Prep: Complete before Tuesday, April 21

Estimated prep time: ~2 hours (required materials). One optional deep dive on the full transformer architecture. All materials are free.

The big picture: In Module 3 you replaced HOG with MobileNetV2 embeddings and showed that learned features capture semantic similarity that hand-crafted descriptors miss. The retrieval worked — similar images landed close together in 1,280-dimensional space. But you stored the embeddings in a numpy array and computed cosine similarity by brute force against every vector in the dataset.

That approach does not scale. A dataset with 10,000 images means 10,000 dot products per query. A dataset with a million means a million. Vector databases like ChromaDB solve this by indexing embeddings so that queries return approximate nearest neighbors without scanning every vector. This week you move from brute-force to indexed retrieval, add quantitative evaluation (precision@k), and run the HOG comparison one more time to lock in the difference.

The readings also cover a conceptual gap: how transformers work. You've been using neural networks as black-box feature extractors. The attention mechanism reading explains what's happening inside — the same mechanism that powers vision transformers (ViT), which are replacing CNNs in many production systems.

Attention in Transformers, Visually Explained Required Video ~25 min

3Blue1Brown • YouTube

Transformers aren't just for text. Vision Transformers (ViT) use the same attention mechanism to process image patches. This video explains how attention works: each element generates a query, every other element generates a key, and the dot product between them determines where the model focuses. The visual style makes the matrix operations intuitive without requiring linear algebra. NLP students are watching this same video — the mechanism is identical across modalities.

Watch: Attention in Transformers →

Key concept to lock in: Attention lets a model decide what to focus on rather than processing everything equally. In a vision transformer, an image patch of a dog's ear can "attend" to the patch containing the dog's nose even if they're on opposite sides of the image. CNNs like MobileNetV2 build this receptive field gradually through stacked layers; transformers get it in a single attention step.
Transfer Learning with Keras and Deep Learning Required Read ~20 min

PyImageSearch • pyimagesearch.com

You've been using MobileNetV2 with include_top=False to extract embeddings. This article explains why that works: the early layers of a pre-trained CNN learn generic features (edges, textures, colors) that transfer across tasks, while the final classification layers are task-specific. Removing the top and using the remaining activations as a feature vector gives you a general-purpose image encoder. The article covers both feature extraction (freeze all layers) and fine-tuning (unfreeze some layers) — your lab uses feature extraction.

Read: Transfer Learning with Keras →

Computer Vision: Algorithms and Applications, §7.1.3–7.1.4: Feature Matching and Large-Scale Retrieval Required Read ~20 min

Richard Szeliski • 2nd Edition (2022) • Free online

Szeliski covers how feature matching works: comparing descriptors across images, nearest-neighbor search, and the ratio test for filtering false matches. The large-scale matching section (§7.1.4) explains why brute-force search doesn’t scale and how indexing structures (kd-trees, hashing) enable retrieval over large image collections — the same problem vector databases solve.

Read: Szeliski §7.1.3–7.1.4 →

Key concept to lock in: Cosine similarity measures the angle between two vectors; Euclidean distance measures the straight-line gap. For normalized embeddings (vectors with unit length), these two metrics produce identical rankings. MobileNetV2 embeddings are not normalized by default — which is why ChromaDB's default L2 distance and cosine similarity may rank results differently.

  1. Your retrieval system returns the 5 most similar images for any query. How would you explain to a non-technical user why a particular image was returned? Can you point to specific visual features, or is the model's reasoning opaque?

Computer Vision Module 4: From Local Search to Vector Database Retrieval

How this lab works: This is a “spec box” lab. You write the code from scratch using the cell specifications below. Each cell lists what it must do, what inputs it takes, and what outputs it must produce. You’re encouraged to use Claude, ChatGPT, or another LLM to help you write the code — paste the spec box and ask the model to implement it. The code structure is intentional: working through it yourself will deepen your understanding of vector databases and similarity search. No “training wheels” scaffold here.

The Task

In the readings and in class, you learned how deep learning models like MobileNetV2 encode images as high-dimensional embeddings, and how vector databases like ChromaDB efficiently index and retrieve embeddings by similarity. This lab upgrades your Module 3 retrieval system: instead of searching in-memory numpy arrays with cosine similarity, you’ll build a persistent, queryable vector database and evaluate retrieval quality using precision@k metrics.

You’ll build a query-by-image system that:

  • Extracts 1280-dimensional embeddings using MobileNetV2 (no classification head)
  • Stores embeddings in a ChromaDB collection with metadata
  • Retrieves similar images by querying the embedding space
  • Evaluates retrieval performance with precision@k
  • Compares MobileNetV2 embeddings against HOG features from Module 2
  • Analyzes failure cases and cross-class confusion patterns

By the end, you’ll have a live retrieval system and insights into which visual categories the model confuses.

Dataset

You may use the same CIFAR-10 images from Module 3 (200 images: 50 per class from 4 classes), or substitute your own set of 200 images organized by class. If using CIFAR-10:

  • Load via tensorflow.keras.datasets.cifar10
  • Use the training set; select one stratified sample of 50 images from 4 classes (e.g., truck, automobile, cat, dog)
  • Resize all images to 224 × 224 (MobileNetV2 input size)
  • Apply keras.applications.mobilenet_v2.preprocess_input()

If using your own images, store them in a folder structure like data/class1/, data/class2/, etc., and load via cv2.imread() or PIL.

Part 2: Build the ChromaDB Collection

# Cell 4: Create ChromaDB Collection

What it does: Create an ephemeral ChromaDB client, initialize a collection, and add all 200 embeddings with metadata.

Takes in: embeddings, labels, class_names

Must produce: client (ChromaDB client), collection (ChromaDB collection named “image_embeddings&rdquo)

Notes:

  • Create an ephemeral client: client = chromadb.EphemeralClient()
  • Create a collection: collection = client.get_or_create_collection(name="image_embeddings")
  • Add embeddings and metadata: 
    collection.add(
    ids=[str(i) for i in range(200)],
    embeddings=embeddings.tolist(),  # Convert numpy array to list
    metadatas=[{"label": labels[i], "index": i} for i in range(200)]
)
  • Print collection.count() to confirm all 200 embeddings were added

Markdown cell: In Module 3, you stored embeddings in a numpy array and computed cosine similarity with a dense matrix operation. Now you’re storing them in ChromaDB. What does a vector database add? (Hint: think about scale, persistence, and query speed.)

# Cell 5: Build Retrieval Function and Display Results

Takes in: collection, embeddings, images, labels

Must produce: function find_similar(query_idx, n=5) and visualizations for one query from each of the 4 classes

Notes:

  • Function signature: def find_similar(query_idx, n=5):
  • Query the collection using the query image’s embedding: results = collection.query(query_embeddings=[embeddings[query_idx].tolist()], n_results=n+1) (n+1 to exclude the query itself later)

  • Return the top-n result indices, labels, and distances
  • Visualization: matplotlib figure with query image on the left, retrieved images in a row to the right. Label each result with its class and distance/similarity score. Use a 1-row, n+1-column layout.
  • Call the function for one query image from each of the 4 classes (use indices that are typical examples, e.g., 12, 62, 112, 162)

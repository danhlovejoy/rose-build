Deliverable 1: Module 6 Lab Lab

Overview

In the readings and in class, you learned how function calling lets an LLM decide when to use external tools, and how the ReAct pattern structures that decision as a reason-then-act loop. This lab implements both. You will define three tools (calculator, date/time, knowledge base), declare them as schemas Gemini can read, build the agent loop that routes tool calls and feeds results back, and test the system on multi-turn conversations and edge cases.

The routing analysis in Cell 6 is where the design decisions become visible. A well-described tool gets called when it should and ignored when it shouldn't. A poorly described tool gets called for everything or nothing. The quality of your tool descriptions determines the quality of your agent.

What to Submit

  • A completed Colab notebook (.ipynb) saved to your GitHub repo at week7/lab6_agents.ipynb.
  • All cells executed with visible outputs (don't clear outputs).
  • Each code cell preceded by a markdown cell with your LLM prompt and fix note.
  • No API keys or secrets anywhere in the notebook.
  • Submit the GitHub link in Canvas. If it's not in the repo, it doesn't exist for grading.
Start this in Thursday lab. We'll build the agent loop together: defining tool schemas, wiring the function-call detection, and debugging the conversation history format. This is the most technically complex lab of the semester — Thursday lab time is essential. Come with Cells 1–3 attempted (setup, tool functions, tool schemas).

Deliverable 2: Module 6 Demo Demo

Overview

A three- to five-minute live demo during Tuesday's class (May 5, 5:30 PM). Your notebook is your demo — no separate slides required. Show the agent working, explain the architecture, and make a live modification.

What to Cover

  1. Architecture overview (30 seconds): Show the tool declarations and explain the agent loop. How does the model decide whether to use a tool?
  2. Tool-using turn: Run a question that triggers a tool. Show the tool-call log. Walk through the flow: user message → model decides to call tool → tool executes → result fed back → final response.
  3. Direct answer: Run a question that does NOT trigger a tool. Show that the model answered from its own knowledge.
  4. Multi-turn context: Show a conversation where the model uses a previous answer in a new tool call (e.g., "take that number and multiply it by 3").
  5. Live modification: Change something — a tool description, a knowledge base entry, or ask a question designed to confuse the routing. Re-run and explain what changed. This is worth 25 points on the demo rubric.

Format

Share your screen and run your notebook live. The interactive chat mode (bonus cell) is ideal for demos — it lets you take questions from the audience. If you attend via Zoom, have your notebook open and ready to run before your turn.

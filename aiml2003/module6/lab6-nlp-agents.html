The Task

In the readings and in class, you learned that function calling lets an LLM decide when to use external tools instead of generating an answer from its own parameters. The model doesn't run the tools itself — it outputs a structured request (tool name + arguments), your code executes the tool, and you feed the result back to the model so it can incorporate the answer into its response.

This changes what an LLM can do. Without tools, Gemini cannot tell you today's date, cannot do reliable arithmetic, and cannot look up current information. With tools, it can do all three — as long as you define the tools and wire the execution loop.

You will build a multi-turn chatbot with three tools:

  • Calculator: Evaluates mathematical expressions.
  • Date/Time: Returns the current date and time.
  • Knowledge Base: Looks up information from a small fact dictionary you define.

The chatbot maintains conversation history across turns, decides when to call a tool vs. answer directly, and handles tool results gracefully. By the end, you will have a working agent loop — the same pattern used by ChatGPT plugins, Claude tools, and production AI assistants.

# Cell 2: Define Tool Functions

What it does: Implements three Python functions that serve as the tools your chatbot can call. These are regular Python functions — they know nothing about Gemini. The agent loop in Cell 4 will call them when the model requests a tool.

Takes in: datetime, math from Cell 1.

Must produce:

  • calculator(expression: str) -> str: Takes a mathematical expression as a string (e.g., "2 + 3 * 4"), evaluates it safely, and returns the result as a string. Use Python's eval() with a restricted namespace that only allows math operations (no builtins, no imports). Return an error message string if the expression is invalid.
  • get_datetime() -> str: Returns the current date, time, and day of the week as a formatted string (e.g., "Tuesday, April 29, 2026, 3:45 PM CDT").
  • knowledge_lookup(topic: str) -> str: Looks up a topic in a predefined dictionary of 10–15 facts. Topics should be things Gemini might get wrong or not know (e.g., current course information, made-up facts for testing, or very recent events). Return the fact if found, or "No information available for: {topic}" if not found. Use case-insensitive matching.
  • TOOL_FUNCTIONS: a dictionary mapping tool names to their Python functions: {"calculator": calculator, "get_datetime": get_datetime, "knowledge_lookup": knowledge_lookup}.
  • Printed output: test each function once and print the result.

Notes:

  • The calculator function must be safe. Use eval(expression, {"__builtins__": {}}, {"math": math}) to prevent code injection. This allows expressions like "math.sqrt(144)" but blocks "__import__('os').system('rm -rf /')".
  • The knowledge base is intentionally small. The point is to show that the model can retrieve facts it doesn't have in its parameters. Include at least one fact about this course (e.g., "The instructor is Dan Lovejoy") and one fact that contradicts common knowledge (to test whether the model trusts the tool over its own training).
Markdown cell: Why is it important to restrict what eval() can execute? What could happen if you passed user input directly to an unrestricted eval()? 
# Cell 3: Define Tool Schemas for Gemini

What it does: Creates the tool declarations that tell Gemini what tools are available, what arguments they take, and what they return. Gemini uses these schemas to decide when to call a tool and how to format its request.

Takes in: genai.types from Cell 1.

Must produce:

  • tools: a list of genai.types.Tool objects (or the equivalent dict format), one for each tool function. Each tool declaration must include:
    • name: matches the key in TOOL_FUNCTIONS (e.g., "calculator").
    • description: a clear sentence explaining when to use this tool. Gemini reads this to decide whether to call it.
    • parameters: a JSON Schema object describing the function's arguments (type, description, required).
  • Print: the tool declarations in a readable format so you can verify the schema is correct.

Notes:

  • The schema format for google-genai uses genai.types.FunctionDeclaration wrapped in genai.types.Tool. Consult the Gemini function calling docs for the exact syntax.
  • The get_datetime tool takes no parameters. Declare it with an empty parameters schema or omit the parameters field entirely.
  • The tool description is critical. A vague description like "does math" will cause Gemini to call the tool unpredictably. A precise description like "Evaluates a mathematical expression and returns the numeric result. Use this for any arithmetic, algebra, or math operation" gives the model clear decision criteria.
Markdown cell: The model decides whether to call a tool based on the tool's description. How would you test whether your descriptions are good enough? What would happen if two tools had overlapping descriptions?

Part 2: The Agent Loop

# Cell 4: Build the Agent Loop

What it does: Implements the core agent loop: send a user message to Gemini with tool declarations, check if the response contains a tool call, execute the tool, send the result back to Gemini, and return the final text response. This is the heart of the lab.

Takes in: client, MODEL_ID, tools, TOOL_FUNCTIONS.

Must produce:

  • chat_history: a list that stores the full conversation (user messages, model responses, tool calls, and tool results). Initialize as empty.
  • SYSTEM_INSTRUCTION: a string that tells Gemini its role and behavior. Include: "You are a helpful assistant with access to tools. Use the calculator for any math. Use get_datetime for time questions. Use knowledge_lookup for factual questions about specific topics. If a tool returns an error, explain the error to the user. Always tell the user when you used a tool to get information."
  • agent_turn(user_message: str) -> str: a function that:
    1. Appends the user message to chat_history.
    2. Calls client.models.generate_content() with the model ID, conversation history, tools, and system instruction.
    3. Checks if the response contains a function_call in response.candidates[0].content.parts.
    4. If yes: extracts the function name and arguments, looks up the function in TOOL_FUNCTIONS, calls it, wraps the result in a genai.types.Part.from_function_response(), appends both the model's tool-call response and the function result to history, and calls generate_content() again so the model can produce a final text answer.
    5. If no: the model answered directly without tools. Extract the text.
    6. Appends the final model response to chat_history.
    7. Returns the text of the final response.
  • Test: call agent_turn("What is 247 * 19?") and print the result. Verify Gemini used the calculator tool.

Notes:

  • The agent loop must handle the case where the model makes a tool call AND the case where it answers directly. Check part.function_call on each part of the response.
  • The google-genai SDK's chat history format expects alternating user and model roles. After a tool call, the tool result goes in as a user role message (or a function role, depending on SDK version). Check the function calling docs for the exact format your SDK version expects.
  • The model may return multiple parts in one response (e.g., text + function call, or multiple function calls). For this lab, handle one tool call per turn. If you want to handle multiple, that's bonus territory.
  • Print a log line each time a tool is called: "[TOOL CALL] calculator('247 * 19') -> 4693". This makes debugging much easier and is great for the demo.
Markdown cell: Trace the data flow for a single tool-using turn. How many API calls did it take? Why does the model need to be called twice — once to decide to use the tool, and again to incorporate the tool's result?

Notes: The model won't be perfect. Some questions are ambiguous (e.g., "How many days until Friday?" could be answered with get_datetime + calculator, or the model might just count in its head). Score generously — the point is to understand the routing patterns, not to hit 100%.

Markdown cell: What was your routing accuracy? Were the errors predictable? For the questions where the model made the wrong routing decision, could you rewrite the tool description to fix it without breaking something else? 
# Cell 7: Error Handling and Edge Cases

What it does: Tests what happens when tools fail or return unexpected results. A robust agent handles errors gracefully instead of crashing.

Takes in: agent_turn.

Must produce:

  • Test these edge cases (fresh conversation for each):
    1. "What is 1 divided by 0?" — the calculator should return an error. Does the model explain the error to the user?
    2. "Tell me about quantum computing." — not in the knowledge base. Does the model handle the "not found" response and fall back to its own knowledge?
    3. "Calculate the answer to life, the universe, and everything, then look it up in the knowledge base." — a request that could chain two tools. Does the model attempt both?
    4. "What is sqrt(-1)?" — the calculator may raise an error for complex numbers. How does the model respond?
  • Print each test case, the tool call(s) made, the tool result, and the model's final response.
  • Print a summary: how many edge cases were handled gracefully vs. resulted in confusing outputs.

Notes: "Handled gracefully" means the model acknowledged the error and gave a useful response. "Confusing" means the model ignored the error, hallucinated an answer, or produced gibberish. If your agent loop crashes on an error instead of returning it to the model, go back to Cell 4 and add a try/except around the tool execution.

Markdown cell: How did the model handle tool errors? Did it acknowledge the failure, or did it try to cover it up? What does this tell you about the model's relationship to its tools — does it trust them unconditionally?

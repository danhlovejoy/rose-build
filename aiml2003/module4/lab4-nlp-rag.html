Module 4 Lab: Chat with a PDF

Build a RAG pipeline that answers questions using a document it has never been trained on.

How this lab works.

Same format as Module 3: each section gives you a specification box describing what a cell must do, what it takes in, and what it must produce. Give the spec to an LLM, generate the code, and fix anything that doesn't match the output contract before moving on.

For each code cell, include a markdown cell with the prompt you used and one sentence describing any fix you made.

The Task

In the readings and in class, you learned that embedding models encode meaning into vectors, and that cosine similarity finds the closest matches. In Module 3 you built a semantic search engine that retrieved similar texts from a fixed dataset. The results were ranked, but they were just quotes from the original data. No synthesis, no reasoning, no answer to a question.

RAG changes that. You still retrieve relevant text chunks using embeddings and cosine similarity, but you pass those chunks to an LLM as context and ask it to generate an answer. The LLM never saw the document during training. Everything it knows comes from what you feed it at query time. This is how production knowledge bases, customer support bots, and document Q&A tools work: retrieval first, generation second.

You will load a PDF, split it into chunks, embed the chunks with the same all-MiniLM-L6-v2 model from Module 3, store them in ChromaDB (a vector database), retrieve the most relevant chunks for a question, and pass them to Gemini with a system prompt that instructs the model to answer using only the provided context. Then you will test what happens when the answer isn't in the document.

Your Document

Choosing a PDF

Pick a PDF that is between 5 and 30 pages of actual text content. Avoid scanned images (the text extractor needs selectable text, not photographs of pages). Good candidates include government reports, technical documentation, open-access research papers, employee handbooks, or product manuals. If you can highlight and copy text from the PDF in your browser, the extractor will work.

If you don't have one in mind, use the OSTP AI Action Plan RFI (January 2025) — a 10-page government document about federal AI policy. It works well because the language is dense enough to produce interesting retrieval results and the content is factual enough to test hallucination behavior.

Upload your PDF to Colab. In the Colab sidebar, click the folder icon, then the upload icon, and drag your PDF in. It will appear at /content/your_file.pdf. Colab files are temporary — if you disconnect and reconnect, you'll need to re-upload. For persistence, upload to your Google Drive and mount it instead.

# Cell 4: Chunk the text

What it does: Splits the full document text into overlapping chunks of roughly equal size. Each chunk becomes one unit that will be embedded and stored in the vector database. Prints the number of chunks, the average chunk length, and the text of the first two chunks.

Takes in: full_text (from Cell 3).

Must produce:

  • CHUNK_SIZE: integer, set to 500. This is the target number of characters per chunk.
  • CHUNK_OVERLAP: integer, set to 100. This is how many characters from the end of one chunk repeat at the start of the next, so that sentences split across a boundary still appear in at least one chunk.
  • chunks: Python list of strings. Name it exactly chunks. Build this by sliding a window of size CHUNK_SIZE across full_text, stepping forward by CHUNK_SIZE - CHUNK_OVERLAP characters each time. Each element is one chunk of text. The implementation is a while-loop:
    1. Start at position 0.
    2. Take full_text[start : start + CHUNK_SIZE] as a chunk.
    3. Advance start by CHUNK_SIZE - CHUNK_OVERLAP.
    4. Stop when start exceeds len(full_text).
  • Printed output: the total number of chunks, the average length in characters (rounded to the nearest integer), and the full text of chunks 0 and 1 (to visually confirm the overlap region).

Notes: The overlap ensures that if a sentence is split between two chunks, the full sentence appears in at least one of them. 500 characters is roughly 75–100 words — long enough to carry meaning, short enough that each chunk is about one paragraph. You can experiment with different values in the bonus section. The last chunk may be shorter than CHUNK_SIZE; that is fine.

Markdown cell: Look at chunks 0 and 1. Can you find the overlapping region? Copy the repeated text into your markdown cell. Why does overlap matter for retrieval? Think about what would happen if a key sentence started in the last 50 characters of one chunk and ended in the first 50 of the next.

Part 2: Build the Vector Store

# Cell 5: Embed the chunks

What it does: Loads the sentence transformer model (the same one from Module 3) and encodes every chunk into a 384-dimensional embedding vector. Prints the shape of the resulting embedding matrix.

Takes in: chunks (from Cell 4).

Must produce:

  • model: a SentenceTransformer object loaded with 'all-MiniLM-L6-v2'. Same model as Module 3.
  • chunk_embeddings: numpy array of shape (len(chunks), 384). Generate by calling model.encode(chunks, show_progress_bar=True). Each row is one chunk's embedding.
  • Printed output: the shape of chunk_embeddings and a note confirming the embedding dimension matches Module 3 (384).

Notes: Encoding time scales linearly with the number of chunks. For a 20-page document with ~80 chunks, expect 3–5 seconds on a GPU runtime. If you're on a CPU runtime, it may take 15–30 seconds.

# Cell 6: Create the ChromaDB collection

What it does: Creates an in-memory ChromaDB database, builds a collection, and inserts all chunks with their embeddings and metadata. Prints the collection count to confirm all chunks were stored.

Takes in: chunks, chunk_embeddings (from Cells 4–5).

Must produce:

  • chroma_client: a ChromaDB client. Create with chromadb.Client() (no arguments — this creates an ephemeral in-memory database).
  • collection: a ChromaDB collection. Create it by calling chroma_client.get_or_create_collection(name="pdf_chunks"). The name is arbitrary but must be consistent across cells.
  • All chunks inserted into the collection using collection.add(). The method requires four arguments:
    • ids: a list of unique string IDs. Use [f"chunk_{i}" for i in range(len(chunks))].
    • documents: the list of chunk texts (chunks).
    • embeddings: the embedding matrix. ChromaDB expects a list of lists, not a numpy array, so convert with chunk_embeddings.tolist().
    • metadatas: a list of dictionaries, one per chunk. Include at least {"chunk_index": i} for each chunk so you can trace results back to their position in the document.
  • Printed output: the collection's item count via collection.count(), confirming it matches len(chunks).

Notes: ChromaDB's in-memory mode means the database lives only in RAM and disappears when the runtime resets. This is fine for a lab — you'll rebuild it by re-running the notebook. The get_or_create_collection method avoids errors if you re-run the cell (it returns the existing collection instead of crashing). Convert embeddings to a list of lists with .tolist() before passing to ChromaDB; numpy arrays cause a type error.

Markdown cell: You just built a vector database. In Module 3 you did similarity search by computing cosine similarity against a numpy array in memory. What does a vector database add? Think about what would happen if your document were 10,000 pages instead of 10.

Part 3: Retrieve Relevant Chunks

# Cell 7: Query the vector store

What it does: Defines a retrieval function that takes a natural language question, encodes it into an embedding, queries ChromaDB for the most similar chunks, and returns them as context. Runs three test queries and prints the retrieved chunks for each.

Takes in: model, collection (from Cells 5–6).

Must produce:

  • retrieve(question, n_results=3): a function. Name it exactly retrieve. It takes a question string and an optional number of results. Inside:
    1. Encode the question: q_embedding = model.encode([question]).tolist().
    2. Query the collection: results = collection.query(query_embeddings=q_embedding, n_results=n_results).
    3. Return results (a dictionary with keys 'documents', 'distances', 'metadatas', and 'ids').
  • Three test queries: write three questions about your document's content. At least one should be a factual question with a specific answer in the text, and at least one should be a question the document probably doesn't answer.
  • Printed output for each query: the question, followed by the top 3 retrieved chunks. For each chunk, print its ID, its distance score (lower is more similar in ChromaDB's default metric), and the first 200 characters of the chunk text.

Notes: ChromaDB's .query() returns distances, not similarities. By default it uses L2 (Euclidean) distance: lower numbers mean closer matches. The query_embeddings parameter expects a list of lists (one list per query), which is why you pass [question] to model.encode() and then call .tolist(). The results dictionary nests lists inside lists: results['documents'][0] gives the list of document texts for the first (and only) query.

Markdown cell: Look at the three retrievals. For the factual question, did the correct chunk appear in the top 3? For the question the document probably can't answer, what did the retriever return anyway? A retriever always returns something — it doesn't know whether the results are actually relevant. This is where generation becomes necessary.

Part 4: Ask the LLM

# Cell 8: Build the RAG prompt

What it does: Defines a function that constructs a prompt for Gemini by combining retrieved chunks with a system instruction and the user's question. The system instruction tells the model to answer using only the provided context and to say "I don't have enough information to answer that" if the context doesn't contain the answer.

Takes in: Nothing directly. This cell defines the prompt template.

Must produce:

  • build_rag_prompt(question, context_chunks): a function. Name it exactly build_rag_prompt. It takes a question string and a list of chunk strings. It returns a single string containing the full prompt. The prompt must have three clearly labeled sections:
    1. System instruction: Tell the model it is a document assistant. It must answer questions using only the provided context. If the answer is not in the context, it must say so explicitly. It must not make up information. It should quote the relevant passage when possible.
    2. Context: The retrieved chunks, numbered and separated by blank lines. Format: --- Chunk 1 ---\n{chunk_text}\n\n--- Chunk 2 ---\n{chunk_text}, and so on.
    3. Question: The user's question, on its own line after the context.
  • Printed output: call the function with one of your test questions and the corresponding retrieved chunks from Cell 7. Print the full assembled prompt so you can see exactly what Gemini will receive.

Notes: The system instruction is where chain-of-thought prompting lives. A good instruction tells the model to first identify which chunk contains the relevant information, then extract the specific passage, then formulate the answer. This stepwise framing reduces hallucination because the model must ground each step in the context. Keep the instruction under 150 words — long instructions dilute the signal.

Markdown cell: Read your assembled prompt. Could a human answer the question from the context alone? If not, the retriever didn't find the right chunks, and no amount of prompt engineering will fix that. RAG fails at the retrieval step more often than the generation step.

Part 5: Test the Boundaries

# Cell 10: Hallucination test

What it does: Sends three questions to the RAG pipeline that the document definitely cannot answer. Evaluates whether the system correctly refuses to fabricate information or whether it hallucinates. Compares the RAG response against a direct Gemini call (no context) on the same question.

Takes in: ask, client (from Cells 9, 2).

Must produce:

  • Three questions that your document cannot possibly answer. These should be factual questions with verifiable answers in a domain completely outside your document's scope. For example, if your document is about AI policy, ask about the boiling point of nitrogen, the population of a specific city, or the release date of a movie.
  • For each question, two responses:
    1. RAG response: Call ask(question). This goes through retrieval and generation.
    2. Direct response: Call Gemini directly with just the question (no context, no system prompt): client.models.generate_content(model='gemini-2.0-flash', contents=question).
  • Printed output: for each question, display both responses side by side. Label them clearly as "RAG (with context)" and "Direct (no context)." After each pair, note whether the RAG system correctly refused to answer.
  • hallucination_count: an integer. Count how many of the three RAG responses fabricated an answer instead of declining. Print a summary: "Hallucination rate: X/3".

Notes: A well-designed RAG system should refuse to answer questions outside its context. If yours doesn't, the system instruction in Cell 8 needs work. This is a real test of your prompt engineering: the model's default behavior is to answer everything, and your system prompt must override that default. The direct Gemini call serves as a control — it shows what the model would say without your RAG constraints.

Markdown cell: Report your hallucination rate. If the RAG system refused to answer all three, your system prompt is doing its job. If it hallucinated on one or more, identify what went wrong. Did the retriever return chunks that the model misinterpreted as relevant? Or did the model ignore the "only use provided context" instruction? These are different failure modes with different fixes.

Module 4: Basic RAG

Readings & Prep: Complete before Tuesday, April 21

Estimated prep time: ~2 hours (required materials). One optional deep dive for those who want the full transformer architecture walkthrough. All materials are free.

The big picture: In Module 3 you built a semantic search engine that found similar sentences by comparing embedding vectors. The retriever worked — it returned relevant results for a given query. But retrieval alone doesn't answer a question. It returns ranked passages, not synthesized explanations.

RAG (Retrieval-Augmented Generation) closes that gap. You retrieve the most relevant passages from a document, inject them into a prompt as context, and ask an LLM to generate an answer grounded in that context. The LLM never saw the document during training. Everything it knows comes from what you feed it at query time. This is how production document Q&A systems, customer support bots, and enterprise knowledge bases work.

This week's readings cover three things: how the attention mechanism inside transformers actually works (the engine behind every model you've used so far), how RAG stitches retrieval and generation together, and how to set up ChromaDB as the vector store between them.

Attention in Transformers, Visually Explained Required Video ~25 min

3Blue1Brown • YouTube

You've been using transformer-based models since Module 3 (the sentence transformer) and since Module 1 (Gemini itself). This video explains what's happening inside: the attention mechanism that lets a transformer decide which parts of the input to focus on when producing each part of the output. 3Blue1Brown's visual style makes the matrix operations intuitive without requiring linear algebra. Watch this first — the vocabulary it introduces (query, key, value, attention head) will appear in readings and class discussions for the rest of the semester.

Watch: Attention in Transformers →

Key concept to lock in: Attention is a learned lookup. Each word generates a query ("what am I looking for?"), and every other word generates a key ("here's what I contain"). The dot product between query and key determines how much attention each word pays to every other word. This is why transformers can connect "it" to "the cat" even when they're 50 words apart — unlike TF-IDF or simple RNNs, attention has no distance penalty.

Read: Retrieval-Augmented Generation →

Key concept to lock in: RAG is not fine-tuning. Fine-tuning changes the model's weights permanently. RAG leaves the model untouched and changes the prompt dynamically based on what's retrieved. This means RAG can answer questions about documents that didn't exist when the model was trained — which is the entire point.

Chunking Strategies for LLM Applications Required Read ~15 min

Pinecone • pinecone.io/learn

Before you can retrieve relevant passages, you have to decide how to split a document into pieces. This guide covers the tradeoffs: fixed-size chunks (simple, predictable, sometimes splits sentences), sentence-level chunks (preserves grammar, varies wildly in size), and recursive splitting (tries to keep paragraphs intact). Your lab uses fixed-size chunking with overlap; this reading explains why that's a reasonable default and when you might want something more sophisticated.

Discussion Prep

These questions will frame our Tuesday class discussion. You don't need to write answers, but come prepared to talk about them.

  1. RAG retrieves chunks and passes them to an LLM as context. The LLM generates an answer from that context. What happens when the retriever returns the wrong chunks? Is the LLM likely to notice, or will it confidently generate an answer from irrelevant context?
  2. Chunk size creates a tradeoff: small chunks are precise but lack context; large chunks carry context but dilute the relevant passage. For a 200-page legal contract, would you chunk by paragraph, by section, or by fixed character count? Why?
  3. The 3Blue1Brown video shows how attention lets a transformer connect distant words. How does this relate to why sentence transformers produce better embeddings than TF-IDF for the retrieval step of RAG?
  4. A company builds a customer support bot using RAG over their product documentation. The bot answers questions accurately 90% of the time and hallucinates 10% of the time. Is 90% good enough to deploy? What factors would change your answer?
